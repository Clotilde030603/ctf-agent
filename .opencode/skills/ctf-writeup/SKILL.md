# CTF Writeup

## What I do

- Turn rough solve notes into a readable writeup.
- Explain the bug, weakness, or logic clearly.
- Include only the evidence and commands that mattered.
- Keep exploit or solver code short and reproducible.

## When to use me

Use this after the challenge is solved or when you want a clean partial-progress handoff.

## Structure

1. Challenge overview.
2. Initial observations.
3. Key vulnerability or insight.
4. Exploitation or solving steps.
5. Final script or command.
6. Flag and short takeaway.

## Rules

- Do not hide failed paths if they explain the final insight.
- Avoid giant code dumps when a short excerpt is enough.
- Keep commands and outputs accurate.
- If something was inferred, say so explicitly.
